# OpenAI Wrapper WebUI 需求分析

## 1. 背景
当前 openai-wrapper 已完成后端代理能力（FastAPI + OpenAI 兼容接口），核心配置依赖手动修改 `config.yaml`。现在需要补一个更实用的 WebUI：不仅能在线保存配置，还要尽量用**表单化方式**降低编辑门槛；同时保留 **raw mode**，方便直接修改原始 YAML。

## 2. 核心目标
WebUI 的核心目标：

1. 在网页上以**表单方式**查看、添加、修改 openai-wrapper 配置
2. 对适合枚举的字段提供**下拉选择**
3. 对列表型配置（如 upstreams、aliases）支持**增删改**
4. 同时提供 **Raw Mode**，允许直接编辑原始 YAML
5. 保存前做校验，避免写坏配置

## 3. 配置对象范围
首版 WebUI 重点覆盖这些核心配置：

### 3.1 Upstreams
每个 upstream 支持：
- name
- base_url
- api_key
- enabled
- kind / type（如 ollama、sglang、openai-compatible 等；若项目已定义枚举，则用下拉）
- models（如有）
- 其他必要字段

要求：
- 支持新增 upstream
- 支持编辑 upstream
- 支持删除 upstream
- 支持启用/禁用

### 3.2 Routing
- 默认 upstream
- 路由规则列表（若当前配置结构已有）

要求：
- 可编辑默认值
- 若是枚举/可选 upstream，优先用下拉选择

### 3.3 Global Chat Overrides
- 支持查看/编辑 default / force / remove 等字段
- 对布尔、数字、字符串字段尽量用合适控件

### 3.4 Global Extra Body
- 支持键值形式编辑
- 首版可以允许用简化 JSON/YAML 文本框处理复杂对象

### 3.5 Aliases
每个 alias 支持：
- alias 名
- target model
- target upstream
- alias overrides
- alias extra_body

要求：
- 支持新增 alias
- 支持编辑 alias
- 支持删除 alias
- target upstream 优先下拉选择

## 4. 交互模式

### 4.1 Form Mode（默认）
默认进入表单模式，按模块展示配置：
- Upstreams
- Routing
- Global Overrides
- Extra Body
- Aliases

特点：
- 可视化编辑
- 列表项支持新增/删除
- 枚举型字段使用下拉
- 布尔值用开关/复选框
- 长文本用 textarea

### 4.2 Raw Mode
支持切换到 Raw Mode：
- 直接编辑完整 YAML
- 适合高级用户做复杂调整
- 保存时同样做 YAML 校验

### 4.3 双模式关系
建议：
- Form Mode 与 Raw Mode 操作同一份配置数据
- 从后端读取配置后，先转成结构化数据供 Form Mode 使用
- 切到 Raw Mode 时展示当前 YAML 文本
- 若在 Raw Mode 修改后保存成功，重新解析并刷新 Form Mode

## 5. 用户场景

### 场景 1：新增一个 upstream
用户在 WebUI 中点击“新增 upstream”，填写名称、base_url、类型，保存即可。

### 场景 2：修改默认路由
用户在 Routing 区域把默认 upstream 改成另一个已存在 upstream，下拉选择后保存。

### 场景 3：新增 alias
用户点击“新增 alias”，选择目标 upstream，填写 target model 和 alias 名，保存。

### 场景 4：高级用户直接改 YAML
用户切换到 Raw Mode，手改 YAML 后保存。

## 6. 功能需求

### 6.1 页面入口
- 提供 `/ui` 入口
- 单页面应用式体验即可，不必拆多页面

### 6.2 配置读取
- 页面加载时读取当前 `config.yaml`
- 后端返回：
  - 原始 YAML 文本
  - 解析后的结构化配置 JSON

### 6.3 表单编辑
- 根据结构化 JSON 渲染配置表单
- 支持列表配置增删改
- 对已知选项字段使用下拉选择
- 支持基础前端校验（必填项、空值、类型）

### 6.4 Raw Mode 编辑
- 可查看和编辑 YAML 原文
- 支持“从表单同步到原文”或“保存后统一以后端为准”

### 6.5 配置保存
- 提交时，后端统一进行合法性校验
- 校验通过后写回 `config.yaml`
- 校验失败时返回明确错误

### 6.6 辅助能力
建议带上：
- 重新加载配置
- 重置当前未保存修改
- 保存结果提示
- 错误提示区域

## 7. 首版不做
- 多用户协作编辑
- 配置变更历史/版本回滚
- 登录鉴权
- 拖拽式可视化流程编排
- 运行时热更新控制台
- 模型测试 playground（先不抢这个范围）

## 8. 约束
- 继续使用 FastAPI 技术栈
- 不引入重型前端框架也可以，但如果纯原生 JS 写表单太乱，可接受轻量结构化实现
- 部署应保持简单，Docker 下可直接访问
- 不影响现有 `/v1/*` API

## 9. 技术方向建议
建议采用以下思路：
- FastAPI 提供 HTML + JSON API
- 前端可以用原生 JS 实现模块化表单
- 若数据绑定太繁琐，可引入轻量方案，但不建议单独起大型前端工程
- 后端新增接口：
  - `GET /api/ui/config`：返回 YAML + 结构化 JSON
  - `POST /api/ui/config`：接收结构化 JSON 或 raw YAML，校验并保存

## 10. 风险点
- 配置结构可能比预想更灵活，纯表单无法覆盖所有边角字段，所以必须保留 Raw Mode
- 不同 upstream 类型字段可能不完全一致，表单需允许保留扩展字段或至少不破坏未知字段
- 双模式同步需要小心，避免覆盖用户未预期的数据

## 11. 推荐结论
首版 WebUI 应做成：
- **默认表单模式**：可增删改、下拉选择、模块化编辑
- **保留 Raw Mode**：直接编辑 YAML
- **统一由后端校验和保存**

这样既不至于简陋成一个大文本框，也不会重到变成完整前端平台。
