# OpenAI Wrapper WebUI 开发设计

## 1. 设计目标
为现有 FastAPI 项目增加一个实用型 WebUI，用于在线管理 `config.yaml`。

本次设计目标不是只做纯文本编辑器，而是：
- 默认提供 **表单化配置编辑**
- 支持列表项的新增/删除/修改
- 对合适字段提供下拉选择
- 同时提供 **Raw Mode** 直接编辑 YAML
- 保存时统一做校验

## 2. 总体方案
采用 **FastAPI + 服务端模板 + 原生 JS** 的轻量方案。

### 2.1 模式划分
页面包含两种编辑模式：

1. **Form Mode**
   - 面向常规配置操作
   - 模块化编辑 upstreams、routing、aliases 等

2. **Raw Mode**
   - 面向高级用户
   - 直接编辑完整 YAML

### 2.2 数据流
后端在读取配置时，同时返回：
- 原始 YAML 文本
- 结构化 JSON 数据

前端：
- Form Mode 基于结构化 JSON 渲染
- Raw Mode 展示 YAML 文本
- 保存时可提交：
  - `mode=form` + JSON 数据
  - `mode=raw` + YAML 文本

后端统一负责：
- 结构校验
- YAML 序列化/解析
- 保存文件

## 3. 目录设计
建议新增文件：

```text
app/
  routers/
    ui.py
  templates/
    ui.html
  static/
    ui.css
    ui.js
```

## 4. 配置处理策略

### 4.1 后端作为配置真源
为避免前端自行拼 YAML 出错：
- Form Mode 下，前端只编辑 JSON 结构
- 保存时发给后端
- 后端再转成 YAML 写入文件

### 4.2 Raw Mode 直接提交文本
- 前端提交完整 YAML 字符串
- 后端 `yaml.safe_load()` 校验
- 校验通过后写入文件

### 4.3 未知字段保留策略
这是个关键点。

建议首版采用：
- 后端读取完整 YAML -> 转 Python dict
- Form Mode 只编辑已支持字段
- 对暂未建表单的字段，尽量保留原始结构，不要无脑丢弃

实现建议：
- 后端保存 Form Mode 时，基于原始配置对象做“定向更新”，而不是重建整个配置对象

这样即使未来 `config.yaml` 多了没做 UI 的字段，也不容易被表单模式抹掉。

## 5. API 设计

### 5.1 页面入口
#### `GET /ui`
返回 WebUI 页面。

### 5.2 读取配置
#### `GET /api/ui/config`
返回：
```json
{
  "path": "/path/to/config.yaml",
  "raw": "upstreams:\n  ...",
  "data": {
    "upstreams": [],
    "routing": {},
    "global_chat_overrides": {},
    "global_extra_body": {},
    "aliases": {}
  },
  "meta": {
    "upstreamKinds": ["ollama", "sglang", "openai-compatible"]
  }
}
```

说明：
- `raw`：Raw Mode 使用
- `data`：Form Mode 使用
- `meta`：下拉选项、枚举提示等

### 5.3 保存配置
#### `POST /api/ui/config`
请求体建议：
```json
{
  "mode": "form",
  "data": { ... }
}
```
或：
```json
{
  "mode": "raw",
  "raw": "upstreams:\n  ..."
}
```

返回：
```json
{
  "ok": true,
  "message": "Config saved successfully",
  "raw": "...最新 YAML...",
  "data": { ...最新 JSON... }
}
```

保存失败：
```json
{
  "ok": false,
  "message": "YAML parse error: ..."
}
```

## 6. Form Mode 页面结构
建议单页按模块分区：

### 6.1 顶部工具栏
- 模式切换：Form / Raw
- 重新加载
- 保存配置
- 状态提示

### 6.2 Upstreams 模块
每个 upstream 渲染成卡片：
- name（文本）
- kind/type（下拉）
- base_url（文本）
- api_key（密码框/文本框）
- enabled（开关）
- 其他字段（必要时扩展）
- 删除按钮

模块底部：
- 新增 upstream 按钮

### 6.3 Routing 模块
- default upstream（下拉）
- 路由规则列表（若存在）
  - path/matcher
  - target upstream
  - 删除按钮
- 新增 rule 按钮

### 6.4 Global Chat Overrides 模块
可分成三块：
- default
- force
- remove

其中：
- 简单键值可表单化
- 复杂对象可用 textarea(JSON/YAML 小块编辑)

### 6.5 Global Extra Body 模块
- 先用 textarea 编辑 JSON/YAML 片段
- 不强行拆得太细

### 6.6 Aliases 模块
每个 alias 用卡片形式：
- alias 名
- target model
- target upstream（下拉）
- overrides（可简化成 textarea/结构区）
- extra_body（textarea）
- 删除按钮

模块底部：
- 新增 alias 按钮

### 6.7 Raw Mode 模块
- 一个大 textarea
- 展示完整 YAML
- 允许直接编辑

## 7. 前端实现策略

### 7.1 状态管理
前端维护一个全局状态对象，例如：
```js
state = {
  mode: 'form',
  raw: '',
  data: {},
  original: {}
}
```

### 7.2 渲染方式
使用原生 JS 按模块渲染 DOM：
- `renderUpstreams()`
- `renderRouting()`
- `renderAliases()`
- `renderRawEditor()`

### 7.3 交互逻辑
- 新增项：向数组/对象插入默认值，再重新渲染
- 删除项：从状态中移除，再重新渲染
- 修改项：直接更新状态
- 切模式：
  - Form -> Raw：把当前结构序列化或使用最近后端返回 raw
  - Raw -> Form：若 raw 改过，建议提示“需先保存后再同步到表单”

### 7.4 首版双向同步建议
为了降低复杂度，首版可以这样处理：
- Form Mode 改动只影响 `state.data`
- 切到 Raw Mode 时，不自动把 Form 改动实时转成 YAML
- 点击保存后，由后端返回最新 `raw + data`
- 保存成功后再同步两边状态

这样少踩很多双向同步坑。

## 8. 后端实现细节

### 8.1 新增路由文件 `app/routers/ui.py`
包含：
- `/ui`
- `/api/ui/config` GET
- `/api/ui/config` POST

### 8.2 配置文件路径
固定使用项目根目录下 `config.yaml`。
不允许用户自定义路径，避免任意文件写入风险。

### 8.3 YAML 解析与保存
- 使用 `yaml.safe_load()` 校验 raw
- Form Mode 保存时，将更新后的 dict 用 `yaml.safe_dump(..., sort_keys=False, allow_unicode=True)` 输出
- 使用临时文件 + `os.replace()` 原子替换

### 8.4 表单保存时的合并策略
为避免丢失未知字段：
- 读取原始配置对象 `original_config`
- 仅覆盖 UI 管理的顶层字段：
  - `upstreams`
  - `routing`
  - `global_chat_overrides`
  - `global_extra_body`
  - `aliases`
- 其他未纳入 UI 的字段原样保留

## 9. 样式建议
因为你要的是“像点样子”的表单 UI，但别太重，所以建议：
- 左右留白的单页布局
- 模块卡片式分区
- 顶部固定工具栏
- 表单项统一 label + input
- 列表项用浅边框卡片包起来
- Raw Mode 用等宽字体 textarea

## 10. 依赖建议
建议确保依赖中包含：
- `Jinja2`
- `PyYAML`

如果当前 requirements 里没有，就补上。

## 11. 开发步骤
1. 新增 `ui.py` 路由
2. 实现读取配置接口（返回 raw + data + meta）
3. 实现保存接口（支持 form/raw 两种模式）
4. 新增 `ui.html`
5. 新增 `ui.css`
6. 新增 `ui.js`
7. 注册路由并本地验证

## 12. 验收标准
完成后至少满足：

1. 打开 `/ui` 能看到表单化配置页面
2. 可新增/编辑/删除 upstream
3. 部分枚举字段可通过下拉选择
4. 可新增/编辑/删除 alias
5. 可切换到 Raw Mode 直接编辑 YAML
6. 保存前后端均有基础校验
7. 非法 YAML 或非法结构不会覆盖原配置
8. 原有 `/v1/*` API 不受影响

## 13. 推荐结论
最终实现方向：
- **主模式：表单编辑器**
- **辅模式：Raw YAML 编辑器**
- **后端做配置合并、校验、落盘**

这版就比较对味了：
不是简陋文本框，但也没重到要单独搞一套前端工程。
